Understanding requirement changes and RCRCRC heuristic

RCRCRC heuristic by Karen N. Johnson helps identify areas that need testing when software changes:

  • Recent - What testing around new areas of code should I think about?
  • Core - What essential functions or features must continue to work?
  • Risky - What features or areas of code are inherently more risky?
  • Configuration Sensitive - What code is dependent on environment settings?
  • Repaired - What code has changed to address defects and potentially created issues?
  • Chronic - What code typically breaks features that need to be tested?

RCRCRC heuristic for regression testing

When requirements change, use this heuristic to identify areas that need testing:

Recent 🆕

What testing around new areas of code should I think about?

  • New priority dropdown functionality
  • Color-coding implementation

Core ⭐

What essential functions or features must continue to work?

  • Task addition still functions
  • List display remains intact

Risky ⚠️

What features or areas of code are inherently more risky?

  • Multiple priority changes
  • Large number of tasks

Configuration-sensitive 🔧

What code is dependent on environment settings?

  • Browser compatibility
  • Screen sizes
  • Accessibility settings

Repaired 🔨

What code has changed to address defects and potentially created issues?

  • Previous bug fixes still work
  • No old issues resurface

Chronic 🔄

What code typically breaks features that need to be tested?

  • Data persistence
  • User interface stability

How to apply RCRCRC

  1. Identify the change (e.g., adding priority feature)
  2. For each RCRCRC category:
    • List relevant test scenarios
    • Prioritize based on risk
    • Execute tests systematically
  3. Document findings and any new risks discovered

How to apply RCRCRC - detailed guide

Step 1: Identify the change

  • Review requirement changes (e.g., V1 to V2 transition)
  • List all new features (Priority dropdown, color coding)
  • Identify impacted existing features (Task list display)

Step 2: Apply RCRCRC categories

Example: Adding priority feature

Category Test Focus Priority
Recent Priority dropdown, color display High
Core Simple task addition High
Risky Multiple priority tasks Medium
Configuration Browser compatibility Medium
Repaired Previous bugs Low
Chronic Data persistence Medium

Step 3: Create a test plan

  1. Organize tests:
    • Group by RCRCRC category
    • Prioritize based on risk level
    • Create test cases or test charters for each scenario
  2. Execute tests:
    • Start with high-priority items
    • Document results systematically
    • Track issues found
  3. Document findings:
    • Record test results
    • Note any new risks discovered
    • Share insights with team

💡 Pro tips:

  • Keep your test plan as a living document - update it as you discover new risks and insights
  • Collaborate with developers to understand technical impacts
  • Consider automated tests for repeatable scenarios
  • Keep track of time spent on each category
