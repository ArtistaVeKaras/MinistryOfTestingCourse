# Conducting Exploratory Testing
The goal is to find issues that may not be covered by pre-existing test cases, so you need to come up with different ways 
to test the feature. As software features are implemented, and user stories become ready for exploratory testing, 
it’s only logical to base exploratory testing sessions on new stories or changed features. Good exploratory testing deals 
with unexpected risks, and for this we need to look beyond the current piece of work. On the other hand, we can’t cast 
the net too widely, because testing would lack focus. A good perspective for investigations that balances wider scope 
with focus is around user capabilities. Features provide capabilities to users to do something useful, or take away user 
capabilities to do something dangerous or damaging. A good way to look for unexpected risks is not to explore features, 
but related capabilities instead.

# How to make it work
To get to good capabilities for exploring, brainstorm what a feature
allows users to do, or what it prevents them from doing. When exploring user stories, try to focus on the user value part 
(‘In order to…’) rather than the feature description (‘I want …’). If you use impact maps for planning work, the third level
of the map (actor impacts) are a good starting point for discussing capabilities. Impacts are typically changes to capabilities. 
If you use user story maps, the top-level item in the user story map spine related to the current user story is a nice 
starting point for discussion.

Good exploratory testing requires good note-taking skills. It looks good and I found this bug might be enough. 
But what are you missing out on by not taking notes to support your summary? 
- What information have you left undiscovered? 
- What did you observe that you didn’t document? 
- What haven't you shared?

Regardless of your note-taking tool these four simple words will keep your notes organised:

- Problems
- Questions
- Ideas
- Praise

Grab a physical notepad and pen or open up an application that allows you to type notes, such as Google Docs, Word or Evernote.
Pick something to test. Maybe the latest deploy includes a story that needs your attention. Set a timer for 30 minutes.
Start exploring! Document in real-time what you’re thinking and observing.

Try to categorise each note as either a problem, question, idea or praise. Don't worry if they don’t fall into one of those categories, 
just leave the categorisation blank. Stop exploring at 30 minutes.

# A Holistic Look At Usability Testing 

It can be thought of as being comprised of five quality components:

* **Learnability**: The design should be intuitive. How easy is it for users to accomplish
basic tasks the first time they use the product? Do they make any errors? Do they get frustrated with the application?

* **Efficiency**: Once users have learned the design, how quickly can they perform tasks? If they struggle with what should be a straightforward task, why is this? Can the number of clicks a user has to make to complete a task be reduced?

* **Memorability**: When users return to the design after a period of not using it, how easily can they reestablish proficiency?

* **Errors**: How many errors do users make, how severe are these errors, and how easily can they recover from the errors?

* **Satisfaction**: How pleasant is it to use the design? Does the user like using the application?

# Six thinking hats
- White hat - what facts do we know and what information do we need? 
- Yellow hat - what are the positives, values and benefits? What occurs when we look at the situation with optimism? 
- Black hat - what are the problems and risks? Potentially the most powerful hat but you need to be careful not to over-focus on this one. 
- Red hat - what do you feel about the situation? Do you have any hunches, likes, dislikes, loves or hates? 
- Green hat - what are the possibilities? Do we have any new ideas, concepts, perceptions or alternatives? 
- Blue hat - what do we think about this? It is also where we control the techniques process and ensure the guidelines are followed.

### How to use testing personas in practice
Testing personas can add focus to exploratory testing sessions. By using personas, testers can step into the mindsets of specific user types, helping them think beyond typical scenarios and find issues real users might face. This approach lets testers adapt their testing style based on each user’s needs, leading to more practical, user-centred results.

* `Set the stage for each test`: Before you start, assign a persona to the session. This way, you’re testing as if you were that user. For instance, if you’re testing for the casual user, check if error messages are clear and if too many options might be confusing.

* `Test realistic workflows`: Focus on workflows that match how each persona would actually use the product. Ask yourself:
   -  What does this persona want to do with this feature?
   -  Is anything in the design confusing or slowing them down?

* `Learn and improve`: After testing with each persona, review what you found. Did the casual user get stuck? Did the advanced user find the feature too slow? Use these findings to make simple, focused improvements.

An example: Testing personas for Facebook’s remember me feature
In lean testing, you focus on the value to the customer, and personas help you pinpoint that value. Let’s look at how personas could be applied to a common feature like “Remember Me” in a login system.

* `Persona 1`: Busy professional
Goal: Save time by not having to log in repeatedly.
Challenges: They have little patience for errors.
Test Focus: Ensure “Remember Me” keeps them logged in seamlessly, without frequent re-authentication.

* `Persona 2`: Privacy-conscious user
Goal: Wants security without hassle.
Challenges: Worried about personal data being saved.
Test Focus: Ensure that “Remember Me” doesn’t bypass security requirements when switching devices or browsers.

In regulated environments, it's crucial to use clear, precise, and scenario-specific language in test reports and documentation to avoid ambiguity. This involves avoiding generalized statements and instead specifying the conditions and scenarios under which tests were conducted. 

**Examples of unsafe language:**
- “It works!”
- “All good”
- “No bugs in the product”
- “Passes as Expected”

**Examples of safe language:**
- “It appears to work under the following conditions …”
- “When tested on the following kinds of scenarios …”
- “Using the following versions of tools, scenarios, etc.”

**Tests for special conditions include:**
- Handling data correctly when users do not accept cookies.
- Implementing disaster recovery in both production and development systems.
- Validating accessibility aspects like sufficient contrast, alt-text for images, screen reader support, and navigation aids.
- Testing for backup, rollback, access control, network security, and processes.
- Ensuring role-based access controls, such as preventing the same person from issuing and authorizing a payment, and tying business and regulatory rules to the software's access control mechanisms.

### How to approach Risks?

In this module we will show you fhe following
   * how to look for risks
   * how to ask questions and some techniques you can use
   * how to document potential risks
   * how to communicate the findings to others

When approaching new software, there are three key questions testers should ask as we begin to understand and discover what the software can do:

What are the potential risks to the software, which could impact the value for end users or customers? 
Can we test to discover how the software behaves when these risks occur?
Do we need to put in mitigations to manage, reduce or remove these risks?

# Ask useful questions:
Lena suggests using questions like “What would happen if…?” to explore potential risks and reveal assumptions and gaps in understanding. Questioning assumptions and challenging vague areas in the artefacts can lead to discovering hidden risks that might otherwise go unnoticed.

# Spot gaps and assumptions:
Gaps in requirements or unclear assumptions are often a source of risk. Look for missing information or implicit expectations in the artefacts. Lena uses the iceberg analogy to explain this: the explicit requirements are above the waterline, but many hidden assumptions lie beneath. Recognising and addressing these assumptions is crucial for thorough risk analysis.

# Take relevant notes:
Note-taking is one of the most valuable tools for testers. Write down potential risks, assumptions, and follow-up questions as you analyse artefacts. These notes help you stay organised and ensure you don’t overlook important points. You can use them later to guide discussions with stakeholders, developers, or other team members.

# Focus on priorities:
Not all risks are equal. Prioritise areas with the highest potential impact or greatest uncertainty for the context of your project. For example, legal compliance (such as GDPR), user experience, or scalability issues might demand more attention. Identifying these critical areas helps you allocate your efforts effectively.

# Use simple models:
Creating diagrams or sketches can help you visualise how the system works and identify potential risks and failure points. Lena demonstrates how to map out processes like order flows, from the user placing an order to generating a work order or invoice. Simple models make it easier to understand complex systems and highlight areas that need closer examination.

Lesson 3: Six powerful techniques and strategies for effective risk analysis
 	
	1. (Favourite) SWOT Analysis
	Strengths - might include a highly skilled automation team
	Weaknesses -  gaps in required skills, lack of access to specific devices for cross-platform testing. Time constraints, limited resources. 
	Opportunities - employing new automation tools, can involve collaboration with external teams who have expertise,
	Threats - Changing requirements that disrupt established testing plans, reliance on third-party integrations that may not be reliable,
   and limited access to real-world devices for testing.

   When to Use it? - Best suited for planning phases of the project.

	2. (Favourite) Risk Matrix - Impact and likelihood 
	Objective: The goal is to help testers concentrate their efforts by prioritising risks according to their likelihood and potential impact.
		
	How It Works:
	Probability is rated as high, medium, or low based on the chances of the risk occurring
	Likelihood is similarly rated on the extent of potential damage if the risk does occur.
	Map each risk onto a matrix where significant likelihood and high-impact risks occupy the most urgent quadrant.

	3. Scenario Analysis
	Objective: To visualise how the system would react to different risk scenarios, considering key variables such as technological advancements,
    regulatory changes, and other unforeseen factors. Enabling teams to plan 	responses to various challenges, each with its own level of impact and likelihood. 
    They ask questions like, 
		"What if there’s a sudden spike in user traffic?"
		"What if there are network interruptions during critical transactions?"
	
	How scenario analysis works:
	Identify the possible risk events for which the scenarios are supposed to be prepared, such as the above-average concurrent users or unknown data. 
   Predict how the system would respond and prepare corresponding test 	cases, scripts or exploratory sessions to ensure resilience in these scenarios.

	4. (Favourite) Risk Storming
	Objective: To gather diverse perspectives from cross-functional teams on recognising, assessing, and creating emergency plans for potential and current risks.

	How Risk Storming works:
	- Assemble a group of stakeholders from relevant departments.
	- Use a collaborative board or visual tool to map and prioritise risks.
	- Discuss strategies for mitigating risks, with each team member offering insights based on their expertise.

	5. Nightmare Headlines Game
	Objective: To uncover hidden risks by imagining potential negative worst-case scenarios and their outcomes. This exercise brings attention to risks
   that might not be obvious and can help teams think critically about 	the repercussions of failure.

	How the nightmare headline game works:
	- Set the stage: Team members picture some hypothetical headlines related to potential project failures.
	- Gather headlines: Each person lists alarming news headlines, imagining worst-case scenarios (e.g., “Data Breach Exposes User Data” or 
   “System Crash That Disrupts Service During Black Friday Sale”).
	Select a key risk: From the list, pick a risk to focus on based on its potential impact.
	- Ideate causes: Identify root causes behind this risk, such as weak security or system bugs.
	- Define charters: Develop particular testing charters that examine each cause with an emphasis on system resilience and prevention.
